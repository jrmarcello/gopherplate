# ADR-002: Estrategia de Identificadores (UUID v7)

**Status**: Aceito (atualizado de ULID para UUID v7)
**Data**: 2026-03-27
**Autor**: Marcelo Jr

---

## Contexto

A escolha do formato de Identificador Unico e critica para sistemas distribuidos e bancos de dados. Precisamos de IDs que sejam:

- **Unicos globalmente**: sem colisoes entre servicos ou replicas.
- **Performaticos**: boa localidade em indices B-tree para insercoes e range scans.
- **Seguros**: nao expor metricas de negocio (como auto-increment faz).
- **Padronizados**: aderentes a especificacoes IETF para interoperabilidade.

---

## Decisao

Utilizar **UUID v7 (RFC 9562)** como chave primaria para todas as entidades.

### Formato

```text
0193a5e8-7c4d-7b2a-8e1f-3c5d7a9b0e2f
└─────────┬─────────┘└───────┬────────┘
   Timestamp (48 bits)    Randomness (74 bits)
   + version/variant
```

UUID v7 combina um timestamp Unix de 48 bits com 74 bits de aleatoriedade, garantindo ordenacao temporal e unicidade global.

---

## Alternativas Consideradas

| Estrategia | Veredicto | Motivo |
| ---------- | --------- | ------ |
| UUID v4 | Rejeitado | Completamente aleatorio, causa fragmentacao severa em indices B-tree |
| Auto-increment (Integer) | Rejeitado | Expoe metricas de negocio, vulneravel a enumeracao, dificulta sharding |
| ULID | Descontinuado | Nao e padrao IETF, requer CHAR(26) ou tipo customizado no PostgreSQL, dependencia adicional (`oklog/ulid`) |
| Hibrido BigInt + UUID v7 | Considerado | Excelente performance para JOINs pesados (BigInt como PK interna, UUID v7 como ID externo). Recomendado para servicos com alta carga de JOINs. Adiciona complexidade de duas colunas de ID |
| **UUID v7** | **Escolhido** | Padrao IETF (RFC 9562), ordenavel por tempo, tipo nativo `UUID` no PostgreSQL, sem dependencias extras |

---

## Justificativa

1. **Padrao IETF**: UUID v7 e definido na RFC 9562, garantindo ampla interoperabilidade e suporte de longo prazo.
2. **Ordenacao Temporal**: Os 48 bits de timestamp garantem inserces sequenciais em indices B-tree, reduzindo fragmentacao.
3. **Tipo Nativo PostgreSQL**: Coluna `UUID` nativa ocupa 16 bytes (vs 26 bytes CHAR para ULID), com validacao e indexacao otimizadas pelo engine.
4. **Zero Dependencias Extras**: Utiliza `github.com/google/uuid` que ja e dependencia padrao do ecossistema Go.
5. **Geracao Descentralizada**: Nao requer sequencia no banco de dados, ideal para sistemas distribuidos.

---

## Consequencias

### Positivas

- Menor fragmentacao de indices comparado a UUID v4.
- Tipo nativo `UUID` no PostgreSQL: 16 bytes, validacao automatica, indexacao otimizada.
- Sem dependencia adicional (`oklog/ulid` removido).
- Padrao IETF com suporte de longo prazo e ampla adocao.

### Negativas

- IDs mais longos em URLs (36 caracteres vs 26 do ULID).
- Requer biblioteca com suporte a UUID v7 (`google/uuid` v1.6+).

---

## Implementacao

### Biblioteca

```go
import "github.com/google/uuid"
```

### Value Object

```go
// domain/user/vo/id.go
type ID string

func NewID() ID {
    return ID(uuid.Must(uuid.NewV7()).String())
}

func ParseID(s string) (ID, error) {
    if _, parseErr := uuid.Parse(s); parseErr != nil {
        return "", ErrInvalidID
    }
    return ID(s), nil
}
```

### Armazenamento (PostgreSQL)

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY,
    ...
);
```

---

## Migracoes

Para servicos existentes que utilizavam ULID, a migracao envolve:

1. Alterar a coluna de `VARCHAR(26)` para `UUID` (requer conversao de dados existentes).
2. Atualizar o Value Object `vo.ID` para gerar UUID v7.
3. Remover a dependencia `oklog/ulid/v2`.

Para novos servicos baseados neste boilerplate, nenhuma migracao e necessaria.
